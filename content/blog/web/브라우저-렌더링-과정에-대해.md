이번 포스트엔 브라우저 렌더링 과정에 대해 적어보려고 한다.

최근 면접 준비 등 여러가지 이유로 인해 포스팅(정리)이 많이 미뤄졌다.

다시 초심으로 돌아가보자!

## 🖥️ 렌더링 과정

### 1. 사용자가 특정 페이지에 접속하여 HTML 파일을 서버로부터 내려 받으면, 브라우저의 렌더링 엔진은 이를 파싱한다.

사용자가 특정 페이지에 접속하면 브라우저는 해당 서버에 `HTTP GET` 요청을 하여 HTML, CSS, JS 파일을 내려 받고 그 중 HTML 파일에 대해 렌더링을 시작한다.

### 2. HTML 파싱을 진행하면서 DOM 트리를 만든다.

### 3. DOM 트리 생성 중 Link 태그를 만나 StyleSheet를 내려받게 될 경우 CSS 파싱을 통해 CSSOM 트리를 만들게 된다.

CSS(CSS Object Model)은 스타일에 관한 tree model 이다.

렌더링 엔진은 인라인 스타일, css 파일 등의 스타일 정보를 종합하여 CSSOM 트리를 만들게 된다.

### 4. 이러한 일련된 과정을 통해 둘을 결합하여 Render Tree를 만들고, 레이아웃 작업을 통해 사용자에게 그려줄 영역을 계산한다.

![image](https://user-images.githubusercontent.com/76273383/178199543-72fdd2fc-a943-48a7-8d4d-cba2f93df4c0.png)

다음 그림은 DOM 트리와 CSSOM 트리가 결합되어 Render Tree를 만드는 사진이다.

Render Tree는 실제 브라우저가 출력할 화면 정보를 담은 트리라고 생각하면 된다.

> 💡 이 과정에서 `display: none`으로 표시된 것은 Render Tree에 포함되지 않는다.

### 5. 위 과정을 진행하면서 자바스크립트를 만나게 되면, 자바스크립트 런타임 환경에 실행권한을 넘겨준다.

Render Tree 생성 중 `<script>` 태그를 통해 자바스크립트 파일을 만나게 되면 렌더링 엔진은 실행권한을
자바스크립트 런타임 환경(JS엔진) 에게 넘겨준다.

이 때 DOM 파싱은 중단된다.

### 6. 모든 과정을 통해 변경된 모든 노드의 수치를 재계산하여 Render Tree를 재생성한다.

위의 모든 과정을 거친 후 레이아웃 변경에 영향을 받은 모든 노드의 수치를 계산하여 Render Tree를 갱신하는 과정을 **Reflow** 라고 한다.

그 후 재생성된 Render Tree를 다시 그리게 되는데 이를 **Repaint**라 한다.

### 7. Repaint가 된 화면을 브라우저에 출력한다.

이렇게 브라우저 렌더링이 완료된다.
